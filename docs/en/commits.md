# Commit Conventions

Conventional commits for consistent history and automated changelogs.

## Impulse

Commit history loses value when messages are written only for the moment instead of for later readers.

- Inconsistent commit messages make history harder to scan
- Release notes become weaker when commit subjects vary too much
- Shared conventions reduce ambiguity during day-to-day work

## Signal

Commit messages shape more than git history. They support releases, changelogs, and team communication.

- A stable structure makes commits easier to search and group
- Conventional types help automation interpret changes reliably
- Clear subjects help future readers understand intent quickly

## Effect

There are several ways to handle commit conventions.

- Informal team habits are easy to start with, but they drift
- A strict local style can work inside one repository, but it does not travel well
- Conventional commits offer a shared format that works across projects and tooling

## Accord

We use conventional commits as the default structure for commit history in Regardio projects.

- Keep subjects short, clear, and written in the imperative
- Use commit types consistently so changes remain legible
- Treat commit history as public project documentation

## Action

Use the format, types, and examples below when writing commits.

### Format

```text
<type>(<scope>): <subject>
[optional body]
[optional footer]
```

- **type**: Category of change (required)
- **scope**: Area affected (optional)
- **subject**: Imperative mood, max 100 chars (required)

### Types

| Type | Description | Changelog |
|------|-------------|----------|
| `feat` | New feature | Features |
| `fix` | Bug fix | Bug Fixes |
| `docs` | Documentation only | - |
| `style` | Formatting | - |
| `refactor` | Code change (no fix/feature) | - |
| `perf` | Performance improvement | Performance |
| `test` | Tests | - |
| `build` | Build/dependencies | - |
| `ci` | CI configuration | - |
| `chore` | Other changes | - |
| `BREAKING CHANGE` | Breaking API change | Breaking Changes |

### Examples

```bash
# Simple
feat: add user authentication
fix: resolve login redirect loop

# With scope
feat(auth): implement OAuth2 flow
fix(api): handle null response gracefully

# With body
fix: prevent race condition in data sync

The previous implementation could cause data corruption.
This adds a mutex lock for sequential processing.

Closes #123

# Breaking change
feat!: redesign authentication API

BREAKING CHANGE: auth.login() now returns Promise.
Update call sites to use async/await.
```

### Best Practices

#### Imperative Mood

```bash
# Good
feat: add validation for email field

# Bad
feat: added validation for email field
```

#### Atomic Commits

Each commit = single logical change. Multiple changes = multiple commits.

#### Reference Issues

```bash
fix: correct calculation of total price

Closes #456
Refs #123, #789
```

### Enforcement

Commit messages are validated by [Commitlint](./toolchain/commitlint.md) via Git hooks.

### Changelog Generation

Changelogs are generated by `ship-production`. Commit subjects become public changelog entries.

## Essence

This convention gives project history a structure that serves both readers and tooling.

- Changes stay easier to understand over time
- Changelog generation can rely on commit intent
- Teams share a common language for describing work
